<decisions_made>

COSTOUNITARIO EDITABLE COMO OVERRIDE (NO REEMPLAZA EL CALCULO):
- El usuario puede tipear directamente costounitario para override manual.
- Si despues cambia conversion/porcentajemerma/precio, se recalcula y pisa el override (ultimo cambio gana).
- Razon: comportamiento minimo y predecible. Si el usuario quiere fijar un valor manual, lo tipea AL FINAL. No hace falta un flag explicito de "manual vs calculado".

PORCENTAJEMERMA: CONVERSION SOLO EN UI, DECIMAL EN STORAGE:
- Display = decimal Ă— 100; onChange = parseFloat(raw)/100 antes de pasar al state.
- DB y calculo siguen viendo decimal (0.10).
- Razon: cambio aislado en una capa (UI). No toca DB, no toca `calcularCostoUnitarioFila`, no toca `registrarInsumosNuevos`. Cero riesgo de inconsistencias cross-modulo.

PORCENTAJEMERMA NORMALIZA TRAILING ZEROS PERO TOLERA STRINGS EN TRANSITO:
- `parseFloat((v * 100).toFixed(6))` para evitar `0.10001 -> 10.000099999...` al mostrar.
- Si raw input es invalido (ej. "1." mientras tipea), se conserva como string para no bloquear escritura.
- Razon: UX limpia. Sin esto, escribir "1.5" requeriria que cada keystroke intermedio fuera valido.

FILTRO COSTO% â€” SOLO PARA RECETAS (PLATILLOS), NO PARA SUBRECETAS:
- `aplicaFiltroPct = filtroCostoPctActivo && visualizarSubTab === "recetas"`.
- Razon: las subrecetas no tienen precioventa â†’ `costoPctNuevo` siempre es null â†’ filtrar las dejaria todas vacias. El UI tampoco muestra el control en sub-tab Subrecetas.

COSTO_PCT_ALERTA (CARD FOOTER) INDEPENDIENTE DEL UMBRAL DEL USUARIO:
- Color del card footer (verde/ambar/rojo) usa constante fija `COSTO_PCT_ALERTA = 30`.
- Filtro usa state `costoPctUmbral` (configurable).
- Razon: si el usuario pone umbral 5% para filtrar "< 5%", no queremos que TODO se vea rojo. El alerta visual es un indicador global de "este platillo cuesta demasiado", el filtro es una query especifica.

PRECIOVENTA: UNA SOLA QUERY POST-LOOP, NO INLINE EN CADA BLOQUE:
- Backend recolecta todos los platilloIds unicos de cascadingPlatillos + directPlatillos, dispara UN solo `IN (...)` query, aplica el map a ambos arrays.
- Razon: minimiza RTT (un query vs dos), evita duplicacion logica. Pattern matching: `if pxmMap[id] === undefined && row.precioventa != null` para tomar el primero (mismo criterio que `analisis-costos-actions.ts:471 â†’ platillosxmenu[0]?.precioventa`).

FILTRO UI INLINE CON CONTROLS, NO POPOVER NI MODAL:
- Grupo compacto: icono + label + select + input + boton toggle. Todo visible siempre que se este en sub-tab Recetas.
- Razon: la operacion es frecuente y los 3 parametros (umbral, comparador, activo) son pequeĂ±os. Un popover seria mas cliks; un modal romperia el flow.

BOTON ACTUALIZAR NUEVOS COSTOS: HARDCODED `disabled` (NO ELIMINAR HANDLER):
